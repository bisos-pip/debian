** BISOS Python Command-Services for interfacing with debain Systemd.

* Architecture Overview

The system uses a *3-layer seed/plantation pattern*:

** Layer 1: Configuration Definition Layer (~planted-socketSysd.cs~)
   - Defines a ~sysdUnitFileFunc()~ that returns a hardcoded INI-format template:
     #+begin_src ini
     [Unit]
     Description=gunicorn socket
     [Socket]
     ListenStream=/run/gunicorn.sock
     [Install]
     WantedBy=sockets.target
     #+end_src
   - Calls ~systemd_seedInfo.setup()~ to register this metadata into the singleton
   - This is the *seed* - the definition of what should be created

** Layer 2: Metadata Storage Layer (~systemd_seedInfo.py~)
   - ~SystemdSeedInfo~ is a *singleton* class that stores configuration metadata
   - It holds 4 key properties:
     - ~seedType~ - the type of unit (e.g., ~"sysdSysUnit"~, ~"sysdUserUnit"~, or potentially ~"sysdSocketUnit"~)
     - ~sysdUnitName~ - the name/identifier of the unit
     - ~sysdUnitFileFunc~ - a callable that returns the systemd file content as a template string
     - ~examplesHook~ - a hook for providing usage examples
   - Acts as the *information registry* that seeds populate

** Layer 3: Processing/Implementation Layer (~bifSystemd_csu.py~)
   - ~ConfigFile_sysdUnit~ class reads from the singleton:
     - Calls ~systemd_seedInfo.systemdSeedInfo.sysdUnitFileFunc()~ to get the template
     - Writes the result to ~/etc/systemd/system/{name}.socket~ (or ~.service~)
   - ~SysUnit~ class provides systemd management operations:
     - ~__init__(name, type="service")~ - accepts type parameter (defaults to "service")
     - ~serviceFilePath()~ - constructs path as ~/etc/systemd/system/{name}.{type}~
     - ~ensure()~ - writes file, reloads daemon, restarts, and enables
     - ~reload()~, ~start()~, ~stop()~, ~restart()~, ~enable()~, ~disable()~ - systemctl wrappers

** Data Flow

   #+begin_src
   planted-socketSysd.cs → systemd_seedInfo (storage) → bifSystemd_csu.py
#+end_src

* Debian 12 Automatic Symlink Creation by systemd

** When `systemctl enable` is run:
   - It reads the [Install] section of the unit file
   - For ~WantedBy=sockets.target~, it automatically creates a symlink in:
     =/etc/systemd/system/sockets.target.wants/{name}.socket=
   - Pointing to the actual unit file location

** Key Finding from Documentation:

   #+begin_quote
   "A symbolic link is created in the .wants/ or .requires/ directory
   of each of the listed units when this unit is installed by systemctl enable."
   #+end_quote

** This means:
   1. Socket file lives in: =/lib/systemd/system/gunicornSocket.socket=
   2. Systemd automatically manages the symlink: =/etc/systemd/system/sockets.target.wants/gunicornSocket.socket=
   3. The symlink is created by =systemctl enable= (no manual symlink creation needed)
   4. The symlink is removed by =systemctl disable=

** Implication for SocketUnit Implementation:
   - We do NOT need to manually create symlinks with pathlib
   - We do NOT need to manually manage symlink lifecycle in =ensure()= or =remove()=
   - The unit file's [Install] section with ~WantedBy=sockets.target~ is sufficient
   - =systemctl enable/disable= handles all symlink operations automatically
